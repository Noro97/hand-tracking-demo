## Verdict

Shippable — nothing here is a regression risk to what's already merged, and the mechanical gates are all green. The one thing worth fixing before the next real-camera test is the `BBTSessionController` bug: a **persisted** handedness relabel (not just a momentary blip) silently stops counting reps for the rest of the session. ## Findings

### [HIGH] A persisted handedness relabel permanently blocks new reps on that hand — `features/bbtSession.ts:186-198`

`handleGestureStart` gates a *new* rep on `handedness !== this.state.selectedHand` (line 190), but `state.selectedHand` is set once in `start()` and never updated. `trackedHandedness`, meanwhile, DOES get reassigned on a successful relabel-resume (`tryResumeFromCandidate`, line 249: `this.trackedHandedness = rep.candidateHandedness`).

**Failure scenario:** `start('Left', ...)`. Rep 1 begins as `'Left'`, MediaPipe relabels the hand to `'Right'` mid-transfer (exactly what `bbtSession.test.ts:97-124` already tests for one rep) — resume succeeds, `trackedHandedness` becomes `'Right'`, rep 1 finalizes and counts correctly. But if the relabel *persists* (not a one-frame blip — MediaPipe keeps calling this physical hand `'Right'` from here on), rep 2's `handleGestureStart('Right', 'thumb-index')` hits the `!this.active` branch, checks `'Right' !== state.selectedHand ('Left')` → true → **returns early, silently dropping the gesture**. Every subsequent rep on that hand is ignored for the rest of the session. No test exercises a second rep after a relabel, so this ships unnoticed.

This directly undermines the point of the relabel-tolerance mechanism task 014 built — it survives *one* rep across a relabel but not the session.

**Suggested fix:** when a resume succeeds, also update `this.state.selectedHand` to the new label (not just `trackedHandedness`), since the proximity check already established it's the same physical hand:
```ts
// tryResumeFromCandidate, after the proximity check passes
this.trackedHandedness = rep.candidateHandedness;
this.state = { ...this.state, selectedHand: this.trackedHandedness };
rep.pendingSince = null;
rep.candidateHandedness = null;
rep.lastPoint = candidate.pointer;
```
Add a test: two reps in one session, relabel during rep 1, rep 2 arrives under the *new* label only (no reversion) — must still count.

### [MEDIUM] `FaceEngine` has no equivalent of the false-positive gate `HandEngine` just added — `engine/faceEngine.ts:66-71`

Task 022 fixed a real bug (a face/beard texture misdetected as a hand) by raising `HandEngine`'s confidence thresholds *and* adding a `MIN_HANDEDNESS_SCORE` gate in `lib/recognition.ts`. `FaceEngine` still hardcodes the MediaPipe defaults with no rationale:
```ts
faceMesh.setOptions({ maxNumFaces: MAX_FACES, refineLandmarks: false, minDetectionConfidence: 0.5, minTrackingConfidence: 0.5 });
```
Face Mesh doesn't expose a per-face classification score the way Hands exposes `multiHandedness[i].score`, so the exact task-022 fix doesn't transfer directly — but the detection/tracking confidence thresholds do, and are currently unexamined. Given the class doc comment claims `FaceEngine` "mirrors HandEngine", this asymmetry is worth a deliberate decision (raise the thresholds to match, or document why 0.5 is fine for a single-face use case) rather than an accident of two features shipped at different times.

### [MEDIUM] `engine/handEngine.ts` / `engine/faceEngine.ts` share ~70 lines of near-identical lifecycle code with no shared base

Constructor (ctx acquisition, canvas sizing, `ResizeObserver`), `stop()`, and the `onResults()` header (`mounted` guard → `reportedReady` gate → `ctx.save/clearRect/drawImage/fillRect`) are duplicated near-verbatim:
- `handEngine.ts:88-100` vs `faceEngine.ts:44-56` (constructor)
- `handEngine.ts:129-136` vs `faceEngine.ts:85-92` (`stop()`)
- `handEngine.ts:138-153` vs `faceEngine.ts:94-109` (`onResults` header)
- `HUD_UPDATE_INTERVAL_MS = 100` declared separately in both (`handEngine.ts:37`, `faceEngine.ts:19`)
- The `Camera` construction block (`handEngine.ts:119-126`, `faceEngine.ts:75-82`) is identical.

Worth extracting into a shared base (owning ctx/resize/mounted/ready-flag/HUD-throttle) once a third engine appears or this drifts further — not urgent with just two, but the confidence-threshold inconsistency above is a concrete sign of drift starting already.

### [MEDIUM] Three demo screens duplicate the same shell markup, and it's already drifted once

`BBTSessionDemo.tsx`, `FaceTrackingDemo.tsx`, `HandFilterDemo.tsx` each repeat: the `flex w-full h-screen bg-page` wrapper, `<MobileBlocker/>`, hidden `<video>`, full-bleed `<canvas>`, `{loading && <LoadingOverlay/>}`, and the bottom instruction pill. Concrete proof this already causes bugs: each file defines its own `Row` component —

`components/BBTSessionDemo.tsx:216-221`:
```tsx
const Row: FC<{ label: string; value: string }> = ({ label, value }) => (
  <div className="flex items-center justify-between text-sm py-1">
```
`components/FaceTrackingDemo.tsx:86-91`:
```tsx
const Row: FC<{ label: string; value: string }> = ({ label, value }) => (
  <div className="flex items-center justify-between py-1">
```
The Face screen's version is missing `text-sm` — same component, same purpose, silently smaller/larger font depending which screen you're on.

**Suggested fix:** one shared `Row` in a small HUD-primitives file (or export it from `FilterPicker.tsx`'s module alongside the picker, since that's already shared), and consider a `TrackingScreenShell` wrapper for the repeated video/canvas/loading/instructions structure.

### [LOW] Duplicated Blob-download logic — `features/bbtSession.ts:227-236` vs `hooks/useFrameRecorder.ts:61-71`

Both do the identical `new Blob(...) → URL.createObjectURL → <a> → click → revokeObjectURL` sequence, differing only in the filename. Small (6 lines) but exact duplication; a shared `lib/download.ts::downloadJson(data, filename)` would remove it.

### [LOW] `lib/fixtures.ts` imports a type from `engine/handEngine.ts`, inverting the codebase's usual `engine → lib` direction

```ts
import type { RawHandFrame } from '../engine/handEngine';
```
Every other file in the codebase depends `engine → lib` (both engines import `lib/colors`, `lib/recognition`, `lib/filterRenderers`). This is type-only (no runtime cycle), but it pulls `features/replay.ts` — explicitly documented as headless/no-DOM — into a transitive dependency on the browser-facing `handEngine.ts` just for one type. `RawHandFrame` would sit more naturally in `lib/recognition.ts` or its own `lib/` module, with `engine/handEngine.ts` importing it instead.

### [LOW] Doc/code mismatch — `features/replay.ts:6-10`

```ts
export interface ReplayOptions {
  selectedHand: Handedness;
  /** Session length in ms — should be >= the fixture's recorded duration or the session will time out mid-replay. */
  durationMs?: number;
}
```
`replayFixture` never calls `controller.tick()` — only `.frame()` per fixture frame and `.stop()` once at the end. Since `tick()` is the only place that checks `remainingMs <= 0` and auto-stops (`bbtSession.ts:173-184`), a replay literally cannot "time out mid-replay" as the comment claims. Either call `tick()` at appropriate points during replay (if timeout behavior should be testable) or fix the comment to describe what actually happens (nothing — the fixture just always plays to its end).
