## Предпочтения
- Язык артефактов (`.md` в `.tasks/` и `docs/`, RETRO, STATE, бриф) — русский.

- `.tasks/` коммитится в git как часть истории репозитория.

### 2026-07-05 — Headless-инфраструктура для "agent repairs itself" не должна реимплементировать движок — извлекать и переиспользовать
**From:** задача 017 (rated pending)
**Lesson:** Когда headless replay/тест-харнесс нуждается в логике, которая живёт внутри engine-класса (например, gesture edge-detection в `HandEngine.dispatchGestureEvents`), НЕ копировать/переписывать эту логику в replay-модуле — вынести её в отдельный переиспользуемый класс (`lib/gestureEventDispatcher.ts`), которым пользуются ОБА потребителя. Реимплементация рискует незаметно разойтись с реальным поведением движка, что делает весь смысл "replay воспроизводит реальный баг" ложным. Извлечение — механический рефакторинг (behavior-preserving), но само по себе даёт первое юнит-тестовое покрытие для кода, которое раньше было недостижимо без DOM/canvas.

### 2026-07-05 — Второй реальный потребитель времени → DI-clock оправдан, даже если раньше решили иначе
**From:** задача 017 (rated pending)
**Lesson:** Задача 015 сознательно решила НЕ добавлять injectable-clock в `BBTSessionController` (единственный потребитель — React-хук с реальными часами, `vi.useFakeTimers()` достаточно для тестов). Задача 017 добавила ВТОРОГО реального потребителя (`features/replay.ts`), которому нужно прогонять `Date.now()`-логику по меткам времени ИЗ ФИКСТУРЫ, а не по реальным часам — вызывать `Date.now()` в цикле replay буквально означало бы, что вся временная логика (relabel-grace, debounce) отыгрывается за микросекунды реального времени, не воспроизводя исходный тайминг. Это ровно тот сценарий, который сама задача 015 отметила как условие для пересмотра решения. Мораль: DI-решение, принятое "для одного потребителя", не догма — пересматривать, когда появляется второй потребитель с другими требованиями, а не городить костыль вокруг старого решения.

### 2026-07-05 — Стабильность identity колбэка важнее самого функционала, если колбэк — часть effect-deps
**From:** задача 017 (rated pending)
**Lesson:** `onRawFrame` передаётся в `useHandTracking`'s `useEffect` deps — если бы его identity менялась при переключении "recording" (например, через `useCallback` с `[recording]` в deps), включение/выключение записи пересоздавало бы `HandEngine` и перезапускало камеру — ровно тот баг, который уже был техдолгом для mode-switcher'а (снят в задаче 014). Решение: "recording активен" живёт в `useRef`, читается ВНУТРИ стабильного (`[]`-deps) колбэка; `useState` — только для того, что должно триггерить UI-рендер (кнопка меняет текст). Общий принцип: если колбэк — часть чужого `useEffect` deps, любое состояние, которое он читает, должно жить в ref, а не быть его собственной deps-зависимостью, если только пересоздание эффекта — не то, что реально нужно.

### 2026-07-05 — Чистая логика тестируется синтетическими данными, даже когда весь остальной проект — "webcam app, нельзя верифицировать headless"
**From:** задача 015 (rated pending)
**Lesson:** "Это webcam-проект, нет камеры в headless" — верно для `engine/handEngine.ts` (DOM/canvas/MediaPipe), но НЕ верно для `lib/filters.ts`, `lib/geometry.ts`, `lib/recognition.ts`, `features/bbtSession.ts` — это чистый TypeScript, принимающий примитивы/plain-объекты (`HandObservation`, landmarks-массивы). Не давать общему ограничению проекта ("нет камеры") распространяться на код, который в нём не нуждается. Синтетические landmark/frame-фикстуры полностью детерминированно проверяют gesture/rep-логику, включая регрессии для прошлых relabel-багов — то, что раньше проверялось только чтением кода руками.

### 2026-07-05 — `Date.now()` внутри тестируемого класса → `vi.useFakeTimers()`, не injectable-clock параметр
**From:** задача 015 (rated pending)
**Lesson:** Когда класс с единственным реальным потребителем (React-хук) вызывает `Date.now()` напрямую внутри методов, и его нужно протестировать детерминированно — не рефакторить публичный конструктор под `now: () => number` параметр ради тестируемости. Vitest `vi.useFakeTimers()` + `vi.setSystemTime()` даёт полный контроль над `Date.now()` без изменения production-API. DI-параметр оправдан только если у класса появится второй реальный потребитель, которому нужен другой clock.

### 2026-07-04 — Реальная камера находит баги, которых headless preview не видит — планировать доводку до PR
**From:** задача 014 (rated pending)
**Lesson:** Три реальных бага (multi-gesture firing, both-hands counting, handedness relabel) появились только когда пользователь протестировал задачу 013 на живой камере — ни один не был видим в headless preview (нет камеры). Это ожидаемо для webcam-проекта, но означает: после смены merge всегда планировать быструю итерацию "пользователь тестирует на устройстве → быстрый фикс-раунд", не считать PR готовым к мержу только по headless-верификации + code review. Фиксы такого раунда — это докрутка ТОЙ ЖЕ задачи/PR, а не новая задача с отдельной веткой (см. ниже).

### 2026-07-04 — Баг-фиксы того же нет-ещё-смерженного PR — та же ветка, не новый PR
**From:** задача 014 (rated pending)
**Lesson:** Когда пользователь находит баги в коде, который ещё не смержен (PR открыт, ревью не пройдено), фиксы идут ДОПОЛНИТЕЛЬНЫМИ коммитами в ТУ ЖЕ ветку/PR, а не в новую ветку. Открывать второй PR, стекованный на первый, добавляет merge-order сложность без реальной пользы на одиночном проекте без CI. Правило "один branch на задачу" из `task-pr-flow` не требует нового branch для докрутки уже открытого PR — это тот же таск, просто ещё не done.

### 2026-07-04 — Документированное "done" ≠ закоммиченное
**From:** задача 013 (rated pending)
**Lesson:** `.tasks/done/NNN.md` и `STATE.md` могут описывать задачу как полностью завершённую (с чеклистом ✅ и verification-логом), пока фактический код остаётся некоммиченным в working tree. Перед тем как ветвиться от "чистого" trunk для новой задачи — сверять `git status`/`git log --stat` с тем, что документы утверждают, а не доверять только STATE.md на слово. Если расхождение найдено — коммитить найденную работу ОТДЕЛЬНЫМ коммитом до начала новой задачи (не смешивать чужой backlog с новым feature-диффом), с явным упоминанием в PR/STATE.md.

### 2026-07-04 — Таймер, завязанный на бизнес-логику, не должен зависеть от частоты источника данных
**From:** задача 013 (rated pending)
**Lesson:** Для BBT-сессии нужен точный обратный отсчёт (60 секунд), но MediaPipe frame loop не гарантирует стабильную частоту (может замедлиться, встать при потере кадра). Решение: обратный отсчёт тикает через собственный `setInterval` в React-хуке (`useBBTSession`, ~4 Гц), полностью независимо от `onFrame`/`onState` колбэков движка. `onFrame` используется ТОЛЬКО для накопления pointer-пути (что действительно должно быть привязано к каждому кадру камеры). Это отдельный паттерн от «троттлинга setState» (см. запись про задачу 003) — там речь о частоте UI-обновлений одного и того же источника данных, здесь — о полной развязке доменного таймера от источника данных вообще.

### 2026-05-29 — `gh repo create --source=. --remote=origin --push` за один шаг
**From:** задача 012 (rated 5/5)
**Lesson:** Когда нужно создать remote-репозиторий из локального проекта — `gh repo create <user>/<name> --public --source=. --remote=origin --push --description="..."` делает всё атомарно: создаёт репо на GitHub, добавляет remote, привязывает branch tracking, пушит. Альтернатива (`gh repo create` без `--source`, потом `git remote add`, потом `git push -u origin master`) — три раздельных шага, каждый из которых может тихо упасть.

### 2026-05-29 — Strict TypeScript — это fix-cascade, не точечная правка
**From:** задача 011 (rated 5/5)
**Lesson:** Включение `strict: true` (и тем более `noUncheckedIndexedAccess`) в проекте, где их не было с начала, **никогда не получится сделать одной строкой в `tsconfig.json`**. Surface'тся: отсутствующие `@types/*` пакеты (которые маскировались `skipLibCheck`), implicit `any` в callback'ах, доступы к массивам без guards, `useRef<T>(null)` сигнатуры в strict-режиме. Реалистичная оценка: 5–15 точек правок на средний проект. Включать strict нужно **вместе с готовностью разгрести cascade одним проходом**, а не «попробуем включить».

### 2026-05-29 — Scale-invariant жесты через relative thresholds
**From:** задача 009 (rated 5/5)
**Lesson:** Для webcam-жестов пороги обязаны быть **относительными** к размеру руки в кадре, не абсолютными в пикселях/нормализованных координатах. Иначе тот же физический жест даёт разные значения метрики на разных дистанциях руки до камеры (close = большие, far = маленькие) и жест ломается. Reference-длина: расстояние от запястья (landmark 0) до middle-MCP (landmark 9) — это «скелетный масштаб» руки, инвариантный к ракурсу. Применять для любых будущих жестов (fist-closure, swipe-distance, finger-spread).

### 2026-05-29 — `preview_snapshot` как backup для `preview_screenshot`
**From:** задача 008 (rated 5/5)
**Lesson:** Когда `preview_screenshot` MCP-инструмент таймаутит (флаки рендерер, навигация залипла), переходить на `preview_snapshot` — он строит accessibility-tree через другой code path и обычно работает. Бонус: a11y-tree показывает ВСЕ элементы DOM независимо от z-index/видимости, что иногда полезнее, чем скриншот (например, для проверки HUD'а, который перекрыт loading-overlay'ем).

### 2026-05-29 — git config-блок проверять до `git add`
**From:** задача 007 (rated 5/5)
**Lesson:** Перед задачами на первичный `git commit` (после `git init` без существующей истории) — заранее проверять `git config --get user.email`. Если пусто, поднимать вопрос на этапе предложения задачи, а не упираться в `fatal: unable to auto-detect email` при выполнении. Safety-protocol запрещает агенту править `git config`, поэтому блок ВСЕГДА требует пользовательского ввода. Лучше выяснить это до того, как stage'нули 25 файлов.

### 2026-05-29 — Конкретность в стратегических брифах
**From:** задача 004 (rated 5/5)
**Lesson:** В стратегических документах MVP-планы и success criteria должны быть привязаны к **именованным валидированным артефактам** домена (конкретный clinical test типа BBT, конкретные CPT-коды, конкретные EHR-системы, конкретные конкуренты с ценами), а не к абстрактным «оценкам прогресса» и «возмещению». Это превращает документ из описательного в actionable и сразу выявляет дыры в знаниях о домене.

### 2026-05-29 — Троттлинг setState из MediaPipe-loop
**From:** задача 003 (rated 5/5)
**Lesson:** `hands.onResults` вызывается на каждом кадре камеры (~30 Гц). Если зеркалить состояние в React через `useState` напрямую — будет 30 ре-рендеров в секунду, а данные для HUD меняются раз в секунды. Паттерн: держать «горячее» состояние в `useRef` для отрисовки на canvas; «холодное» зеркало для React-HUD обновлять через ref-таймер (~10 Гц достаточно для UI). Применять везде, где UI-индикатор берёт данные из MediaPipe/getUserMedia-цикла.

### 2026-05-29 — Порядок при удалении импортируемого файла
**From:** задача 002 (rated 5/5)
**Lesson:** При удалении файла, который импортируется другими модулями, *сначала* очистить импорты у потребителей, *потом* удалять файл. Если сделать наоборот, Vite (любой HMR) пытается транспилировать stale-импорт между правками и засоряет логи `Internal server error: Failed to resolve import "..."`. Сборка может пройти, потому что финальное состояние корректно, но dev-сервер показывает ложные ошибки, которые легко принять за реальные.

### 2026-05-29 — Минимальная prep-зачистка перед рефакторами
**From:** задача 001 (rated 5/5)
**Lesson:** Когда пользователь просит «cleanup → рефактор» в одной директиве, делать prep-зачистку отдельной мелкой задачей с *честной* минимальной областью (только то, что реально мертво сегодня, без захвата того, что снимется в 002/003). Это даёт чистый чекпоинт и не дублирует работу.
